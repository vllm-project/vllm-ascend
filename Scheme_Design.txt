# vllm-ascend External DP Nightly 测试框架设计方案

## 1. 背景

当前 vllm-ascend nightly multi-node 测试框架主要覆盖 internal load balancing 或 hybrid load balancing 场景。

DeepSeek-V3.1 文档中的外部负载均衡部署方式与当前 nightly 抽象存在明显差异：

```text
当前 nightly 常见抽象：
  一个 node -> 一个 server_cmd -> 一个 vLLM endpoint

external DP 抽象：
  一个 node -> 多个 local DP rank -> 多个 vLLM endpoint
```

文档中的 external DP 方式大致由两部分组成：

```text
1. launch_online_dp.py
   负责根据 dp_size_local / dp_rank_start / tp_size / port_start 等参数，
   在当前节点上启动多个 vLLM backend。

2. run_dp_template.sh
   负责接收 launch_online_dp.py 传入的参数，
   生成最终 vllm serve 命令。
```

本次设计目标是：在 nightly multi-node 测试框架中新增一套 external DP 测试能力，使其既支持非 PD 分离场景，也支持 PD 分离场景。

---

## 2. 目标

### 2.1 功能目标

新增 external DP 测试框架，支持：

```text
1. generic_dp
   非 PD 分离 external DP 场景。
   多个 vLLM backend endpoint 由 generic DP proxy 做负载均衡。

2. disaggregated_prefill
   PD 分离 external DP 场景。
   多个 prefiller endpoint 和 decoder endpoint 由 PD proxy 做请求转发。
```

### 2.2 YAML 驱动目标

测试配置完全由 YAML 驱动。

YAML 需要同时满足：

```text
1. 可读性强
2. 接近文档中的 launch_online_dp.py + run_dp_template.sh 方式
3. 不保留旧式 server_cmd
4. 支持 config 参数回填到 env 和 server_cmd_template
5. 支持每个节点独立模板
6. 支持 routing 通过序号引用 config/template
7. 支持框架自动生成 DP_RANK / PORT / VISIBLE_DEVICES
8. 支持最终完整命令可打印、可追踪、可 debug
```

### 2.3 测试目标

新增：

```text
1. generic_dp 小规模 e2e case
   - 使用 Qwen/Qwen3-0.6B
   - 2 节点，每个节点启动多个 external DP endpoint

2. disaggregated_prefill 小规模 e2e case
   - 使用 vllm-ascend/DeepSeek-V2-Lite-W8A8
   - 2 节点，prefiller / decoder 分离
   - 覆盖 Mooncake kv_producer / kv_consumer 路径

3. DeepSeek-V3.1 external DP PD 正式拓扑 YAML
   - 按 DeepSeek-V3.1 文档中的 W8A8 2P1D Layerwise external DP PD 拓扑编写
   - 默认只支持 workflow_dispatch 手动触发，不进入第一版定时 schedule

4. 配置解析 / 命令生成 / endpoint 推导 / proxy 命令生成 / result writer UT

5. 独立 external_dp reusable workflow 和 A3 schedule workflow

6. benchmark result JSON 新增 external_dp_topology 字段
```

### 2.4 实现约束

后续编码应遵循以下工程约束：

```text
1. 先明确假设，再编码
2. 只实现 external DP 测试框架需要的最小能力
3. 不顺手重构现有 multi_node 测试框架
4. 每个新增行为都要有可验证的 UT 或 e2e 验收点
5. 代码改动保持外科式，避免 speculative abstraction
6. 优先保证现有 nightly 用例不受影响
```

## 5. YAML 总体结构

最终 YAML 采用如下结构：

```yaml
test_name: "test xxx"
model: "xxx"
request_model: "xxx"  # optional
num_nodes: 4
npu_per_node: 16
cluster_hosts: []  # optional, only for non-K8s/local debug

env_common: &env_common
  ...

config_common: &config_common
  ...

routing:
  ...

config:
  - ...
  - ...

templates:
  - ...
  - ...

benchmarks:
  ...
```

各部分含义：

```text
test_name:
  测试名称。

model:
  vllm serve 使用的模型路径或模型名。

request_model:
  可选字段。
  AISBench / OpenAI API 请求中使用的 model 名称。
  如果未配置，则默认使用顶层 model。
  如果 server_cmd_template 中显式配置了 --served-model-name，
  建议同步配置 request_model，避免请求模型名和服务暴露模型名不一致。

num_nodes:
  当前测试需要的节点数量。

npu_per_node:
  每个节点的 NPU 数量，用于校验 visible_devices 是否越界。

cluster_hosts:
  可选调试字段。
  CI 中默认不配置，框架通过 LWS DNS 解析所有节点 IP。
  如果配置，则 len(cluster_hosts) 必须等于 num_nodes，
  且 cluster_hosts[i] 对应 $NODE_i_IP。

env_common:
  所有节点可复用的基础环境变量。

config_common:
  所有节点可复用的基础测试框架配置，例如 host、port_start、dp_rpc_port。

routing:
  描述当前 external DP 路由类型，以及哪些 config/template 属于 prefiller、decoder 或 worker。

config:
  节点级结构化配置列表。
  config[i] 表示第 i 个节点的 external DP 配置。
  第一版强制 config[i].node_index == i。

templates:
  节点级模板列表。
  templates[i] 与 config[i] 一一对应。
  templates[i] 里定义该节点的 envs 和 server_cmd_template。

benchmarks:
  accuracy / performance 测试配置。
```

---

## 6. YAML 关键设计原则

### 6.1 config 和 templates 通过序号一一对应

规则：

```text
config[0] + templates[0] -> 第 0 个节点的配置和模板
config[1] + templates[1] -> 第 1 个节点的配置和模板
config[2] + templates[2] -> 第 2 个节点的配置和模板
...
```

框架必须校验：

```text
len(config) == len(templates) == num_nodes
config[i].node_index == i
```

如果不满足，直接报错。

第一版不支持 config 顺序和 node_index 不一致的写法，避免 routing 序号、config 下标、node_index 三者产生歧义。

---

### 6.2 routing 使用序号引用 config/template

PD 分离场景：

```yaml
routing:
  type: "disaggregated_prefill"
  groups:
    prefiller: [0, 1]
    decoder: [2, 3]
```

含义：

```text
config[0] + templates[0] 属于 prefiller
config[1] + templates[1] 属于 prefiller
config[2] + templates[2] 属于 decoder
config[3] + templates[3] 属于 decoder
```

非 PD 场景：

```yaml
routing:
  type: "generic_dp"
  groups:
    worker: [0, 1]
```

含义：

```text
config[0] + templates[0] 属于 worker
config[1] + templates[1] 属于 worker
```

---

### 6.3 server_cmd_template 不写 vllm serve

YAML 顶层已经写了：

```yaml
model: "Qwen/Qwen3-0.6B"
```

因此 `server_cmd_template` 只写 `vllm serve` 后面的参数，不写：

```text
vllm serve <model>
```

框架最终自动拼接：

```bash
vllm serve ${MODEL} ${rendered_server_cmd_template}
```

如果需要设置 `--served-model-name`，直接把它作为普通参数写在 `server_cmd_template` 中。

框架不为 `--served-model-name` 提供单独顶层字段，也不对它做特殊处理。

如果配置了 `--served-model-name`，建议同时配置：

```yaml
request_model: "<served-model-name>"
```

这样 AISBench 请求模型名与服务暴露模型名一致。

---

### 6.4 不再保留旧式 server_cmd

不使用：

```yaml
server_cmd: >
  vllm serve ...
```

而是使用：

```yaml
server_cmd_template:
  - --host
  - ${HOST}
  - --port
  - ${PORT}
  - --data-parallel-size
  - ${DP_SIZE}
```

它不是最终命令，而是会被每个 local rank 渲染多次的参数模板。

## 7. YAML 变量体系

YAML 中存在两类变量。

---

### 7.1 集群占位符

这类变量使用 `$XXX` 形式，主要出现在 `config`、`routing`、`envs` 中。

支持：

```text
$LOCAL_IP
$MASTER_IP
$NODE_0_IP
$NODE_1_IP
...
$NODE_{i}_IP，其中 i ∈ [0, num_nodes)
$LWS_WORKER_INDEX
```

示例：

```yaml
dp_address: "$NODE_2_IP"
proxy_host: "$NODE_0_IP"
```

说明：

```text
1. 如果 cluster_hosts 非空，则 $NODE_i_IP 来自 cluster_hosts[i]。
2. 如果 cluster_hosts 为空，则 $NODE_i_IP 由 LWS DNS / 当前 CI 机制解析。
3. $LOCAL_IP 对应当前 LWS_WORKER_INDEX 的节点 IP。
4. $MASTER_IP 默认等价于 $NODE_0_IP。
```

---

### 7.2 命令模板变量

这类变量使用 `${XXX}` 形式，主要出现在 `server_cmd_template` 中。

支持：

```text
${MODEL}
${REQUEST_MODEL}
${HOST}
${PORT_START}
${PORT}
${DP_SIZE}
${DP_SIZE_LOCAL}
${DP_RANK_START}
${DP_RANK}
${LOCAL_RANK}
${TP_SIZE}
${PP_SIZE}
${DP_ADDRESS}
${DP_RPC_PORT}
${VISIBLE_DEVICES}
${NODE_INDEX}
${CONFIG_INDEX}
```

其中：

```text
${MODEL} = YAML 顶层 model，用作 vllm serve 模型名
${REQUEST_MODEL} = YAML 顶层 request_model；如果未配置，则等于 model
${PORT} = port_start + local_rank
${DP_RANK} = dp_rank_start + local_rank
${VISIBLE_DEVICES} = 根据 local_rank 和 tp_size 自动生成
```

## 8. Endpoint 自动推导规则

对每个 `config[i]`，框架根据 `dp_size_local` 展开多个 endpoint。

伪代码：

```python
for local_rank in range(dp_size_local):
    dp_rank = dp_rank_start + local_rank
    port = port_start + local_rank

    visible_devices = ",".join(
        str(device)
        for device in range(
            local_rank * tp_size,
            (local_rank + 1) * tp_size,
        )
    )
```

例如：

```yaml
config[3]:
  node_index: 3
  dp_size: 32
  dp_size_local: 16
  dp_rank_start: 16
  tp_size: 1
  port_start: 7100
```

生成：

```text
local_rank 0  -> dp_rank 16 -> port 7100 -> visible_devices 0
local_rank 1  -> dp_rank 17 -> port 7101 -> visible_devices 1
...
local_rank 15 -> dp_rank 31 -> port 7115 -> visible_devices 15
```

## 11. YAML Schema：DeepSeek-V3.1 正式拓扑要点

正式拓扑 YAML 对应：

```text
config/DeepSeek-V3.1-external-dp-pd.yaml
```

该配置第一版只支持 `workflow_dispatch` 手动触发，不进入定时 schedule。

正式拓扑要求：

```text
1. routing.type = disaggregated_prefill
2. num_nodes = 4
3. npu_per_node = 16
4. routing.groups.prefiller = [0, 1]
5. routing.groups.decoder = [2, 3]

prefiller:
  node 0:
    dp_group = prefill_0
    dp_size = 2
    dp_size_local = 2
    dp_rank_start = 0
    tp_size = 8
    dp_address = $NODE_0_IP

  node 1:
    dp_group = prefill_1
    dp_size = 2
    dp_size_local = 2
    dp_rank_start = 0
    tp_size = 8
    dp_address = $NODE_1_IP

decoder:
  node 2:
    dp_group = decode
    dp_size = 32
    dp_size_local = 16
    dp_rank_start = 0
    tp_size = 1
    dp_address = $NODE_2_IP

  node 3:
    dp_group = decode
    dp_size = 32
    dp_size_local = 16
    dp_rank_start = 16
    tp_size = 1
    dp_address = $NODE_2_IP
```

该 YAML 的 `templates` 需要根据实际 DeepSeek-V3.1 W8A8 2P1D Layerwise 文档补齐：

```text
1. DeepSeek-V3.1 模型路径
2. quantization
3. expert parallel
4. kv-transfer-config
5. layerwise / PD 相关参数
6. speculative / compilation 等实际需要参数
7. prefiller 和 decoder 的差异化 env
```

## 12. 命令生成规则

对每个节点，框架执行：

```text
config[i] + templates[i] -> node i 的启动配置
```

对每个 local rank，框架执行：

```text
1. 计算 LOCAL_RANK
2. 计算 DP_RANK
3. 计算 PORT
4. 计算 VISIBLE_DEVICES
5. 渲染 envs
6. 渲染 server_cmd_template
7. 拼接 vllm serve <model>
8. 启动 subprocess
```

最终命令形式：

```bash
ASCEND_RT_VISIBLE_DEVICES=<visible_devices> \
<rendered envs> \
vllm serve <model> <rendered server_cmd_template>
```

```bash

  --data-parallel-rank 16 \
  --data-parallel-address 10.0.0.2 \
  --data-parallel-rpc-port 12321 \
  --tensor-parallel-size 1 \
  --served-model-name DeepSeek-V3.1 \
  --trust-remote-code \
  --quantization ascend \
  --enable-expert-parallel \
  --kv-transfer-config '{"kv_connector":"MooncakeConnectorV1","kv_role":"kv_consumer","kv_port":"30200","engine_id":"3"}'
```

## 14. 配置校验规则

框架加载 YAML 后必须校验：

```text
1. num_nodes > 0
2. len(config) == num_nodes
3. len(templates) == num_nodes
4. config[i].node_index 唯一
5. 第一版强制 config[i].node_index == i
6. config[i] 与 templates[i] 一一对应
7. routing.type 必须是 generic_dp 或 disaggregated_prefill
8. generic_dp 必须存在 routing.groups.worker
9. disaggregated_prefill 必须存在 routing.groups.prefiller 和 routing.groups.decoder
10. routing.groups 中的 index 必须在 [0, num_nodes) 范围内
11. dp_size_local * tp_size <= npu_per_node
12. dp_rank_start + dp_size_local <= dp_size
13. 同一节点内 port 不冲突
14. 同一节点内 visible_devices 不重叠
15. proxy_node_index 必须在 [0, num_nodes) 范围内
16. 每个 config 必须包含 host / port_start / dp_rpc_port / dp_size / dp_size_local / dp_rank_start / tp_size / dp_address
17. 每个 template 必须包含 envs 和 server_cmd_template
18. backend healthcheck_path 默认 /health
19. 第一版 proxy healthcheck_path 固定 /healthcheck；如后续 proxy 脚本不一致，再在 routing 中增加 proxy_healthcheck_path
20. 如果 cluster_hosts 非空，则 len(cluster_hosts) 必须等于 num_nodes
21. 如果 request_model 未配置，则 request_model 默认等于 model
```

## 20. 风险与规避

### 20.1 端口冲突

风险：

```text
多个 endpoint 使用 port_start 自动递增，可能和已有进程冲突。
```

规避：

```text
1. 启动前检查端口可用性
2. port_start 和 proxy_port 由 YAML 配置
3. 端口冲突时打印 node_index / local_rank / port
```

---

### 20.2 多进程清理失败

风险：

```text
一个节点上启动多个 vLLM 进程，失败后容易残留。
```

规避：

```text
1. 所有进程由 ExternalDPServerManager 记录 PID
2. cleanup 使用 process tree 级别清理
3. finally 中强制 cleanup
4. CI 结束前可做一次兜底 pkill，但要限制匹配范围
```

---

### 20.3 YAML 模板变量错误

风险：

```text
server_cmd_template 中写错 ${XXX}，导致命令生成错误。
```

规避：

```text
1. command_builder 对未知变量直接报错
2. UT 覆盖常见变量
3. 日志打印最终渲染命令
```

---

### 20.4 routing.groups 配置错误

风险：

```text
routing.groups 引用错误 index，导致 proxy backend 列表错误。
```

规避：

```text
1. 加载 YAML 时校验 index 范围
2. disaggregated_prefill 必须同时包含 prefiller 和 decoder
3. generic_dp 必须包含 worker
4. result JSON 中记录 groups 和 endpoint 明细
```

---

### 20.5 request_model 与 served-model-name 不一致

风险：

```text
server_cmd_template 中配置了 --served-model-name，
但 AISBench 请求仍使用 model 字段，导致请求模型名不匹配。
```

规避：

```text
1. 新增 request_model 字段
2. request_model 未配置时默认等于 model
3. 如果模板中包含 --served-model-name 但 request_model 未显式配置，可以打印 warning
```

---

### 20.6 影响现有 nightly

风险：

```text
新框架修改现有 nightly 公共逻辑，影响已有用例。
```

规避：

```text
1. 新增 external_dp 子目录
2. 不修改 RemoteOpenAIServer 行为
3. 不替换现有 multi_node workflow
4. 新增独立 external_dp workflow
```

## 23. 一句话总结

本设计通过 `config[i] + templates[i]` 的方式复刻文档中 `launch_online_dp.py + run_dp_template.sh` 的思想：`config` 负责描述节点级 DP 拓扑，`templates` 负责描述节点级命令模板，`routing.groups` 通过序号声明 endpoint 分组，Python manager 自动生成每个 local rank 的 `DP_RANK / PORT / VISIBLE_DEVICES`，最终渲染出完整 `vllm serve <model>` 命令，并同时支持 generic external DP 和 PD 分离 external DP 两种 external load balancing 场景。
