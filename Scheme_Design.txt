# vllm-ascend External DP Nightly 测试框架设计方案

## 1. 背景

当前 vllm-ascend nightly multi-node 测试框架主要覆盖 internal load balancing 或 hybrid load balancing 场景。

DeepSeek-V3.1 文档中的 external DP / external load balancing 部署方式与当前 nightly 抽象存在明显差异：

```text
当前常见 nightly 抽象：
  一个 node -> 一个 server_cmd -> 一个 vLLM endpoint

external DP 抽象：
  一个 node -> 多个 local DP rank -> 多个 vLLM endpoint
```

文档中的 external DP 部署方式可以理解为：

```text
1. launch_online_dp.py 根据 dp_size_local / dp_rank_start / tp_size / port_start 等参数，
   在当前节点上启动多个 vLLM backend。

2. run_dp_template.sh 接收 launch_online_dp.py 传入的参数，
   生成最终 vllm serve 命令。

3. 外部 proxy 根据 endpoint 列表做请求分发：
   - generic_dp 使用 dp_load_balance_proxy_server.py
   - disaggregated_prefill 使用 load_balance_proxy_server_example.py
```

本方案的目标是在 nightly multi-node 测试框架中新增一套 external DP 测试能力，使其同时支持：

```text
1. generic_dp：非 PD 分离 external DP 场景
2. disaggregated_prefill：PD 分离 external DP 场景
```

当前实现已经落地在：

```text
tests/e2e/nightly/multi_node/external_dp/
```

并新增了独立 workflow：

```text
.github/workflows/_e2e_nightly_external_dp.yaml
.github/workflows/schedule_nightly_external_dp_a3.yaml
```

---

## 2. 设计目标

### 2.1 功能目标

新增 external DP nightly 测试框架，支持：

```text
1. YAML 驱动 external DP 拓扑描述
2. 一个节点启动多个 vLLM backend endpoint
3. 自动推导 local_rank / dp_rank / port / visible_devices
4. 自动渲染 envs 和 server_cmd_template
5. 自动拼接 vllm serve <model>
6. 支持 generic_dp proxy
7. 支持 disaggregated_prefill proxy
8. 支持 AISBench accuracy / performance
9. 支持 benchmark result JSON 输出
10. benchmark result JSON 新增 external_dp_topology 字段
11. 支持 UT 验证配置解析、endpoint 推导、命令生成、proxy 命令生成、结果写入
12. 支持独立 external_dp workflow
```

---

### 2.2 YAML 驱动目标

YAML 需要同时满足：

```text
1. 可读性强
2. 接近文档中的 launch_online_dp.py + run_dp_template.sh 方式
3. 不保留旧式完整 server_cmd
4. config 参数可以回填到 envs 和 server_cmd_template
5. 每个节点有独立 template
6. routing 通过序号引用 config/template
7. 框架自动生成 DP_RANK / PORT / VISIBLE_DEVICES
8. 最终完整命令可打印、可追踪、可 debug
9. AISBench 请求直接使用顶层 model，和现有多节点 nightly 保持一致
10. env 可以引用 config 变量，server_cmd_template 可以用 shell 风格 $VAR 引用已渲染 env
```

---

### 2.3 测试目标

当前包含三类 YAML：

```text
1. generic_dp_smoke.yaml
   - 非 PD external DP 小规模验证
   - 2 节点
   - 每个节点启动多个 backend endpoint
   - 使用 generic DP proxy

2. disaggregated_prefill_smoke.yaml
   - PD 分离 external DP 小规模验证
   - 2 节点
   - node 0 为 prefiller
   - node 1 为 decoder
   - 使用 PD proxy
   - 覆盖 kv_producer / kv_consumer 路径

3. DeepSeek-V3.1-external-dp-pd.yaml
   - DeepSeek-V3.1 external DP + PD 分离正式拓扑
   - 4 节点
   - 2P1D
   - prefiller: node 0 / node 1
   - decoder: node 2 / node 3
   - 默认用于 workflow_dispatch 手动触发
```

## 3. 非目标

第一版不做：

```text
1. 不改造现有 RemoteOpenAIServer 行为
2. 不破坏现有 tests/e2e/nightly/multi_node 用例
```

```text

  |
  v

  |
  v

  |
  v

  |
  v

  |
  v
ExternalDPProxyLauncher
  |
  v
AISBench accuracy / performance
  |
  v
write_benchmark_results_json
  |
  v
benchmark result artifact + logs artifact
```

运行时节点分工：

```text
所有节点：
  1. 加载 YAML
  2. 解析所有 endpoint
  3. 找出当前节点 endpoint
  4. 启动本节点多个 vLLM backend
  5. 等待本节点 backend /health ready

master 节点，即 node 0：
  1. 等待所有 backend ready
  2. 启动 proxy
  3. 等待 proxy /healthcheck ready
  4. 运行 AISBench perf / acc
  5. 写 benchmark result JSON
  6. 写 done signal

非 master 节点：
  1. 启动 backend 后等待 done signal
  2. 收到 done signal 后 cleanup
```

## 6. YAML 总体结构

当前 YAML 采用如下结构：

```yaml
test_name: "test xxx"
model: "xxx"
num_nodes: 2
npu_per_node: 16
cluster_hosts: []     # optional, local debug only

env_common: &env_common
  ...

config_common: &config_common
  ...

routing:
  ...

config:
  - ...
  - ...

templates:
  - ...
  - ...

benchmarks:
  perf:
    ...
  acc:
    ...
```

字段说明：

```text
test_name:
  测试名称，也可作为 benchmark job name fallback。

model:
  vllm serve 和 AISBench/OpenAI API 请求共同使用的模型路径或模型名。
  CommandBuilder 会自动拼接：
    vllm serve <model> <rendered_server_cmd_template>

num_nodes:
  当前测试需要的节点数量。

npu_per_node:
  每个节点的 NPU 数量，用于校验 visible_devices 是否越界。

cluster_hosts:
  可选字段，主要用于非 K8s / 本地 debug。
  CI 中通常不配置，框架通过 LWS DNS 解析所有节点 IP。
  如果配置，则 len(cluster_hosts) 必须等于 num_nodes，
  且 cluster_hosts[i] 对应 $NODE_i_IP。

env_common:
  所有节点可复用的基础环境变量。

config_common:
  所有节点可复用的基础测试框架配置。
  例如 host、port_start、dp_rpc_port。
  backend healthcheck_path 和 ready_timeout 由测试框架后台常量控制，不在 YAML 中配置。

routing:
  描述当前 external DP 路由类型，以及哪些 config/template 属于 worker、prefiller、decoder。

config:
  节点级结构化配置列表。
  config[i] 表示第 i 个节点的 external DP 配置。
  node_index 可作为可读性字段保留；如果出现，必须等于 i，但不会进入 NodeConfig 数据类。

templates:
  节点级模板列表。
  templates[i] 与 config[i] 一一对应。
  templates[i] 定义该节点的 envs 和 server_cmd_template。

benchmarks:
  AISBench accuracy / performance 测试配置。
```

---

## 7. YAML 关键设计原则

### 7.1 config 和 templates 通过序号一一对应

规则：

```text
config[0] + templates[0] -> 第 0 个节点的配置和模板
config[1] + templates[1] -> 第 1 个节点的配置和模板
config[2] + templates[2] -> 第 2 个节点的配置和模板
...
```

框架必须校验：

```text
len(config) == len(templates) == num_nodes
如果 config[i].node_index 存在，则必须等于 i
```

第一版不支持 config 顺序和 node_index 不一致的写法，避免 routing 序号、config 下标、node_index 三者产生歧义。

---

### 7.2 routing 使用序号引用 config/template

PD 分离场景：

```yaml
routing:
  type: "disaggregated_prefill"
  groups:
    prefiller: [0, 1]
    decoder: [2, 3]
```

含义：

```text
config[0] + templates[0] 属于 prefiller
config[1] + templates[1] 属于 prefiller
config[2] + templates[2] 属于 decoder
config[3] + templates[3] 属于 decoder
```

非 PD 场景：

```yaml
routing:
  type: "generic_dp"
  groups:
    worker: [0, 1]
```

含义：

```text
config[0] + templates[0] 属于 worker
config[1] + templates[1] 属于 worker
```

EndpointResolver 会根据 routing.groups 给每个 endpoint 标注 role。

---

### 7.3 server_cmd_template 不写 vllm serve

YAML 顶层已经写了：

```yaml
model: "Qwen/Qwen3-0.6B"
```

因此 `server_cmd_template` 只写 `vllm serve` 后面的参数，不写：

```text
vllm serve <model>
```

框架最终自动拼接：

```bash
vllm serve ${model} ${rendered_server_cmd_template}
```

例如 YAML 中写：

```yaml
server_cmd_template:
  - --host
  - ${HOST}
  - --port
  - ${PORT}
  - --data-parallel-size
  - ${DP_SIZE}
```

最终渲染为：

```bash
vllm serve Qwen/Qwen3-0.6B \
  --host 0.0.0.0 \
  --port 7100 \
  --data-parallel-size 4
```

---

### 7.4 不保留旧式完整 server_cmd

不使用：

```yaml
server_cmd: >
  vllm serve ...
```

使用：

```yaml
server_cmd_template:
  - --host
  - ${HOST}
  - --port
  - ${PORT}
  - --data-parallel-size
  - ${DP_SIZE}
```

`server_cmd_template` 不是最终命令，而是会被每个 local rank 渲染多次的参数模板。

## 8. YAML 变量体系

YAML 中存在两类变量：

```text
1. $XXX
   集群占位符，主要出现在 config、routing、envs 中。

2. ${XXX}
   严格模板变量，可出现在 envs 和 server_cmd_template 中。
3. $VAR
   shell 风格变量，envs 可用它引用 config 派生变量；server_cmd_template 可用它引用已渲染 env。
```

---

### 8.1 集群占位符

使用 `$XXX` 形式。

支持：

```text
$LOCAL_IP
$MASTER_IP
$NODE_0_IP
$NODE_1_IP
...
$NODE_{i}_IP，其中 i ∈ [0, num_nodes)
$LWS_WORKER_INDEX
```

示例：

```yaml
dp_address: "$NODE_2_IP"
proxy_host: "$NODE_0_IP"
```

解析规则：

```text
1. 如果 cluster_hosts 非空：
     $NODE_i_IP 来自 cluster_hosts[i]

2. 如果 cluster_hosts 为空：
     $NODE_i_IP 由 LWS DNS / 当前 CI 机制解析

3. $LOCAL_IP：
     当前 LWS_WORKER_INDEX 对应的节点 IP

4. $MASTER_IP：
     默认等价于 $NODE_0_IP
```

---

### 8.2 config 派生模板变量

使用 `${XXX}` 形式。

当前支持：

```text
${MODEL}
${HOST}
${PORT_START}
${PORT}
${DP_SIZE}
${DP_SIZE_LOCAL}
${DP_RANK_START}
${DP_RANK}
${LOCAL_RANK}
${TP_SIZE}
${PP_SIZE}
${DP_ADDRESS}
${DP_RPC_PORT}
${VISIBLE_DEVICES}
${NODE_INDEX}
${CONFIG_INDEX}
```

含义：

```text
${MODEL}
  YAML 顶层 model，用作 vllm serve 和 AISBench 请求模型名。

${HOST}
  config[i].host。

${PORT_START}
  config[i].port_start。

${PORT}
  port_start + local_rank。

${DP_SIZE}
  config[i].dp_size。

${DP_SIZE_LOCAL}
  config[i].dp_size_local。

${DP_RANK_START}
  config[i].dp_rank_start。

${DP_RANK}
  dp_rank_start + local_rank。

${LOCAL_RANK}
  当前节点内第几个 local rank。

${TP_SIZE}
  config[i].tp_size。

${PP_SIZE}
  config[i].pp_size。

${DP_ADDRESS}
  config[i].dp_address。

${DP_RPC_PORT}
  config[i].dp_rpc_port。

${VISIBLE_DEVICES}
  根据 local_rank 和 tp_size 自动生成。

${NODE_INDEX}
  当前节点 index。

${CONFIG_INDEX}
  当前 config/template 下标。
```

如果 `server_cmd_template` 引用了未知 `${XXX}`，CommandBuilder 应直接报错，不应静默忽略。

### 8.3 env 到 cmd 的 shell 风格变量

渲染顺序：

```text
1. 先渲染 template.envs。
2. envs 可以使用 `${PORT}` 或 `$PORT` 这类 config 派生变量。
3. 再渲染 server_cmd_template。
4. server_cmd_template 可以使用 `$SERVER_PORT` 这类已渲染 env。
```

示例：

```yaml
templates:
  - envs:
      ASCEND_RT_VISIBLE_DEVICES: "$VISIBLE_DEVICES"
      SERVER_PORT: "$PORT"
    server_cmd_template:
      - --port
      - $SERVER_PORT
```

## 9. Endpoint 自动推导规则

对每个 `config[i]`，框架根据 `dp_size_local` 展开多个 endpoint。

伪代码：

```python
for local_rank in range(dp_size_local):
    dp_rank = dp_rank_start + local_rank
    port = port_start + local_rank
    visible_devices = ",".join(
        str(device)
        for device in range(
            local_rank * tp_size,
            (local_rank + 1) * tp_size,
        )
    )
```

例如：

```yaml
config[3]:
  node_index: 3
  dp_size: 32
  dp_size_local: 16
  dp_rank_start: 16
  tp_size: 1
  port_start: 7100
```

生成：

```text
local_rank 0  -> dp_rank 16 -> port 7100 -> visible_devices 0
local_rank 1  -> dp_rank 17 -> port 7101 -> visible_devices 1
...
local_rank 15 -> dp_rank 31 -> port 7115 -> visible_devices 15
```

当前 `ExternalDPEndpoint` 需要携带的信息包括：

```text
config_index
role
dp_group
local_rank
dp_rank
host
bind_host
port
visible_devices
dp_size
dp_size_local
tp_size
pp_size
dp_address
dp_rpc_port
```

## 10. routing 类型

当前支持两种 routing type：

```text
1. generic_dp
2. disaggregated_prefill
```

---

### 10.1 generic_dp

YAML：

```yaml
routing:
  type: "generic_dp"
  proxy_node_index: 0
  proxy_host: "$NODE_0_IP"
  proxy_port: 1999
  proxy_script: "examples/external_online_dp/dp_load_balance_proxy_server.py"
  groups:
    worker: [0, 1]
```

框架行为：

```text
1. 收集 role == worker 的所有 endpoint。
2. 生成 generic DP proxy 命令。
3. proxy 参数使用：
     --dp-hosts
     --dp-ports
```

最终 proxy 命令形式：

```bash
python examples/external_online_dp/dp_load_balance_proxy_server.py \
  --host <proxy_host> \
  --port <proxy_port> \
  --dp-hosts <worker_host_0> <worker_host_1> ... \
  --dp-ports <worker_port_0> <worker_port_1> ...
```

---

### 10.2 disaggregated_prefill

YAML：

```yaml
routing:
  type: "disaggregated_prefill"
  proxy_node_index: 0
  proxy_host: "$NODE_0_IP"
  proxy_port: 1999
  proxy_script: "examples/disaggregated_prefill_v1/load_balance_proxy_server_example.py"
  groups:
    prefiller: [0]
    decoder: [1]
```

框架行为：

```text
1. 收集 role == prefiller 的所有 endpoint。
2. 收集 role == decoder 的所有 endpoint。
3. 生成 PD proxy 命令。
4. proxy 参数使用：
     --prefiller-hosts
     --prefiller-ports
     --decoder-hosts
     --decoder-ports
```

最终 proxy 命令形式：

```bash
python examples/disaggregated_prefill_v1/load_balance_proxy_server_example.py \
  --host <proxy_host> \
  --port <proxy_port> \
  --prefiller-hosts <prefiller_host_0> <prefiller_host_1> ... \
  --prefiller-ports <prefiller_port_0> <prefiller_port_1> ... \
  --decoder-hosts <decoder_host_0> <decoder_host_1> ... \
  --decoder-ports <decoder_port_0> <decoder_port_1> ...
```

## 12. disaggregated_prefill smoke YAML 要点

对应文件：

```text
tests/e2e/nightly/multi_node/external_dp/config/disaggregated_prefill_smoke.yaml
```

用途：


```yaml
test_name: "test DeepSeek-V2-Lite-W8A8 external dp disaggregated_prefill"

env_common: &env_common
  VLLM_USE_MODELSCOPE: "true"
  OMP_PROC_BIND: "false"
  OMP_NUM_THREADS: "10"
  PYTORCH_NPU_ALLOC_CONF: "expandable_segments:True"

config_common: &config_common
  host: "0.0.0.0"
  port_start: 7100
  dp_rpc_port: 12321

routing:
  type: "disaggregated_prefill"
  proxy_node_index: 0
  proxy_host: "$NODE_0_IP"
  proxy_port: 1999
  proxy_script: "examples/disaggregated_prefill_v1/load_balance_proxy_server_example.py"
  groups:
    prefiller: [0]
    decoder: [1]
```

## 14. 命令生成规则

对每个节点：

```text
config[i] + templates[i] -> node i 的启动配置
```

对每个 local rank：

```text
1. 计算 LOCAL_RANK
2. 计算 DP_RANK
3. 计算 PORT
4. 计算 VISIBLE_DEVICES
5. 渲染 template.envs
6. 渲染 template.server_cmd_template
7. ASCEND_RT_VISIBLE_DEVICES 由 template.envs 使用 $VISIBLE_DEVICES 显式渲染
8. 自动拼接 vllm serve <model>
9. 启动 subprocess
```

最终命令形式：

```bash
ASCEND_RT_VISIBLE_DEVICES=<rendered_visible_devices> SERVER_PORT=<rendered_port> \
vllm serve <model> <rendered_server_cmd_template>
```

示例：

```bash
ASCEND_RT_VISIBLE_DEVICES=0 \
SERVER_PORT=7100 \
vllm serve vllm-ascend/DeepSeek-R1-0528-W8A8 \
  --host 0.0.0.0 \
  --port 7100 \
  --data-parallel-size 32 \
  --data-parallel-rank 16 \
  --data-parallel-address 10.0.0.2 \
  --data-parallel-rpc-port 12321 \
  --tensor-parallel-size 1 \
  --trust-remote-code \
  --quantization ascend \
  --enable-expert-parallel \
  --kv-transfer-config '{"kv_connector":"MooncakeConnectorV1","kv_role":"kv_consumer","kv_port":"30200","engine_id":"3"}'
```

## 15. 配置校验规则

框架加载 YAML 后必须校验：

```text
1. num_nodes > 0
2. len(config) == num_nodes
3. len(templates) == num_nodes
4. 如果 config[i].node_index 存在，则必须等于 i
5. config[i] 与 templates[i] 一一对应
6. routing.type 必须是 generic_dp 或 disaggregated_prefill
7. generic_dp 必须存在 routing.groups.worker
8. disaggregated_prefill 必须存在 routing.groups.prefiller 和 routing.groups.decoder
9. routing.groups 中的 index 必须在 [0, num_nodes) 范围内
10. 同一个 config index 不能同时出现在多个 routing group 中
11. dp_size_local * tp_size <= npu_per_node
12. dp_rank_start + dp_size_local <= dp_size
13. 同一节点内 port 不冲突
14. 同一节点内 visible_devices 不重叠
15. routing.proxy_node_index 必须在 [0, num_nodes) 范围内
16. 每个 config 必须包含 host / port_start / dp_rpc_port / dp_size / dp_size_local / dp_rank_start / tp_size / dp_address
17. 每个 template 必须包含 envs 和 server_cmd_template
18. backend healthcheck_path 由框架固定为 /health
19. 第一版 proxy healthcheck_path 固定 /healthcheck
20. backend ready_timeout 由框架后台常量控制
21. 如果 cluster_hosts 非空，则 len(cluster_hosts) 必须等于 num_nodes
```

### 17.1 CommandBuilder

职责：

```text
1. 接收 ExternalDPConfig
2. 接收 ExternalDPEndpoint

5. 渲染 envs
```

渲染规则：

```text
1. ${XXX} 找不到时直接报错

5. ${MODEL} 来自顶层 model

7. envs 可以用 $PORT / $VISIBLE_DEVICES 引用 config 派生变量
8. server_cmd_template 可以用 $SERVER_PORT 引用已渲染 env
9. envs 的值最终都需要转为字符串
```

```text
1. 将 env 和 command list 格式化成可读命令

3. 敏感 env key 需要脱敏
```

敏感 key 包含：

```text

SECRET
```

### 17.3 process utils

当前进程工具包括：


## 24. 风险与规避

### 24.1 端口冲突

风险：

```text
多个 endpoint 使用 port_start 自动递增，可能和已有进程冲突。
```

规避：

```text
1. port_start 和 proxy_port 由 YAML 配置
2. 端口冲突时打印 node_index / local_rank / port
3. 日志中记录最终 endpoint 列表
```

---

### 24.2 多进程清理失败

风险：

```text
一个节点上启动多个 vLLM 进程，失败后容易残留。
```

规避：

```text
1. 所有进程由 ExternalDPServerManager / ExternalDPProxyLauncher 记录 PID
2. cleanup 使用 process tree 级别清理
3. finally 中强制 cleanup
4. start_process 使用 start_new_session=True
```

---

### 24.3 YAML 模板变量错误

风险：

```text
server_cmd_template 中写错 ${XXX}，导致命令生成错误。
```

规避：

```text
1. CommandBuilder 对未知变量直接报错
2. UT 覆盖常见变量
3. 日志打印最终渲染命令
```

---

### 24.4 routing.groups 配置错误

风险：

```text
routing.groups 引用错误 index，导致 proxy backend 列表错误。
```

规避：

```text
1. 加载 YAML 时校验 index 范围
2. disaggregated_prefill 必须同时包含 prefiller 和 decoder
3. generic_dp 必须包含 worker
4. 同一个 index 不能同时出现在多个 group
5. result JSON 中记录 groups 和 endpoint 明细
```

---

### 24.5 请求模型名与服务暴露模型名不一致

风险：

```text
如果 vllm serve 命令显式暴露了不同于顶层 model 的模型名，
但 AISBench 请求仍使用 model 字段，可能导致请求模型名不匹配。
```

规避：

```text
1. 第一版不新增独立请求模型字段，AISBench 固定使用顶层 model。
2. 默认 YAML 不配置额外服务暴露名，让 vLLM 暴露名称与 model 保持一致。
3. 如果后续必须在 vllm serve 命令中显式设置服务暴露名，需要同步确认请求 model 策略。
```

---

### 24.6 日志分散导致排查困难

风险：

```text
多节点、多 endpoint、多进程日志分散，失败时难定位。
```

规避：

```text
1. 每个 endpoint 独立 rank log
2. 每个节点打包 external_dp logs artifact
3. benchmark result JSON 记录 serve_cmd 和 endpoint topology
4. leader pod stream 主流程日志
```

## 25. 当前实现和早期方案的差异

早期方案曾计划拆分为：

```text
endpoint_resolver.py
command_builder.py
proxy_launcher.py
server_manager.py
process_utils.py
result_writer.py
```

当前实际实现更紧凑：

```text
external_dp_config.py
  - config loader
  - placeholder resolver
  - dataclass
  - EndpointResolver

external_dp_utils.py
  - CommandBuilder
  - process utils
  - proxy command builder
  - result writer

test_external_dp.py
  - ExternalDPServerManager
  - ExternalDPProxyLauncher
  - pytest orchestration
```

这样做的优点：

```text
1. 第一版文件更少
2. 逻辑更集中
3. 便于快速 review
4. 不影响现有 multi_node 框架
```

后续如果 external_dp 框架继续扩展，可以再按职责拆分文件。

## 28. 一句话总结

当前 external DP nightly 框架通过 `config[i] + templates[i]` 复刻文档中 `launch_online_dp.py + run_dp_template.sh` 的思想：

```text
config:
  描述节点级 DP 拓扑，例如 dp_size、dp_size_local、dp_rank_start、tp_size、port_start、dp_address。

templates:
  描述节点级 envs 和 server_cmd_template。

routing.groups:
  通过序号声明哪些 config/template 属于 worker、prefiller、decoder。

Python manager:
  自动生成每个 local rank 的 DP_RANK / PORT / VISIBLE_DEVICES，
  渲染 envs 和 server_cmd_template，
  自动拼接 vllm serve <model>，
  启动多个 vLLM backend。

Proxy launcher:
  根据 routing.type 启动 generic_dp proxy 或 disaggregated_prefill proxy。

Result writer:
  输出 benchmark result JSON，
  并新增 external_dp_topology.summary 和 external_dp_topology.endpoints。
```

最终效果是：同一套 YAML 驱动框架同时支持 generic external DP 和 PD 分离 external DP，两类 smoke case 进入 external_dp nightly workflow，DeepSeek-V3.1 完整 external DP PD 拓扑通过 workflow_dispatch 手动触发。
