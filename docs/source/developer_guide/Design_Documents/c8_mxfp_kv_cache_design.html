C8_MXFP KV Cache 存储设计

目标：当 quant_description["kv_cache_type"] == "C8_MXFP" 时，KV cache 本体使用 FP8 E4M3，并额外缓存 K/V 的 E8M0 scale。当前阶段完成 scale cache 的写入和读取，但不把 scale 传给 attention 算子。

GQA KV Cache MXFP8 E8M0 Scale Read Scale

设计原则是复用现有 C8 KV cache 的接入方式，并仿照 sparse C8 的多 tensor cache 模式扩展 K/V scale cache。

存储布局

C8_MXFP 下 KV cache 从二元组扩展为四元组。GQA 场景按 num_kv_heads 存储，不按 num_heads 展开。

Tensor dtype shape 说明
k_cache torch.float8_e4m3fn [num_blocks, block_size, num_kv_heads, head_size] K cache 本体。
v_cache torch.float8_e4m3fn [num_blocks, block_size, num_kv_heads, head_size] V cache 本体。
k_scale_cache float8_e8m0fnu 或 uint8 视图 [num_blocks, block_size, num_kv_heads, head_size / 32] K 的 per-32 元素 scale。
v_scale_cache float8_e8m0fnu 或 uint8 视图 [num_blocks, block_size, num_kv_heads, head_size / 32] V 的 per-32 元素 scale。 
kv_caches[layer_name] = (
    k_cache,
    v_cache,
    k_scale_cache,
    v_scale_cache,
)

关键约束

  • kv_cache_type 必须为 C8_MXFP 才启用该存储路径。
  • head_size 必须能被 32 整除；本阶段不引入 padding。
  • scale 按最后一维每 32 个元素共享一个 E8M0 scale。
  • KV scale cache 与 KV cache 使用相同 slot_mapping 写入。
  • decode 等没有新 K/V 写入的场景，也会在 backend forward() 入口读取 kv_cache[2:4] 并缓存到 backend 属性。
  • attention 算子调用保持现状，暂不把 scale cache 传给 FIA/PA。
  • 当前阶段不考虑 PD 分离和 kv transfer，C8_MXFP 新增 cache tensor 不做 kv transfer 对齐或 producer/consumer 分支。

Scale Cache 读取

scale cache 的读取在 AscendC8MXFPAttentionBackendImpl.forward() 入口完成。这样 decode-only 场景即使没有新的 K/V 写入，也能从 kv_cache 四元组中恢复 scale cache 引用。

def _read_mxfp_scale_cache(self, kv_cache):
    if len(kv_cache) < 4:
        return None, None
    self._init_mxfp_cache_refs(kv_cache)
    return self.key_scale_cache, self.value_scale_cache

读取后的引用保存在 backend 属性中：

self.key_scale_cache = kv_cache[2]
self.value_scale_cache = kv_cache[3]

当前阶段只保证引用可读、可随 block copy/swap 同步维护，后续如果 attention 算子支持 mxfp8 KV 反量化，可在专门的 decode/prefill 路径中消费这两个属性。

内存开销

每个 K 或 V 元素本体为 1 byte FP8，每 32 个元素额外 1 byte scale。因此相对 K/V 本体，scale 额外开销为 1 / 32 = 3.125%。

per_layer_page_bytes =
    2 * block_size * num_kv_heads * head_size
  + 2 * block_size * num_kv_heads * (head_size / 32)

当前不做的内容

  • 不修改 FIA/PA attention 算子调用参数。
  • 不新增 mxfp8 KV cache 反量化消费逻辑；scale cache 只读取到 backend 属性，不参与计算。
  • 不处理 PD 分离、kv transfer producer/consumer 差异和跨实例 KV 搬运。
  • 不支持 hamming sparse KV compression；开关打开时 C8_MXFP backend 会显式报错。
  • 不支持 head_size % 32 != 0 的 padding。
  • 不改 checkpoint scale name remap，因为 K/V scale 是运行时动态生成。

下一步计划

AscendC8MXFPAttentionBackendImpl.forward() 已重写为 MXFP8 attention 路径：动态量化 query，读取已缓存的 K/V scale，并把 query/key/value 的 E8M0 scale 传给 quantized FIA 调用。当前实现依赖 torch_npu runtime 对 FusedInferAttentionScore V5 风格 MXFP8 scale 参数的支持。

目标行为

  • Query 仍保持现有 attention 输入 dtype，或按算子要求扩展为全量化 query 路径。
  • K/V cache 本体从 kv_cache[0] 和 kv_cache[1] 读取，dtype 为 torch.float8_e4m3fn。
  • K/V scale cache 从 kv_cache[2] 和 kv_cache[3] 读取，shape 为 [num_blocks, block_size, num_kv_heads, head_size / 32]。
  • attention 算子调用时同时传入 K/V 本体与 K/V scale，使算子按最后一维每 32 个元素共享一个 E8M0 scale 完成反量化或融合计算。
  • GQA 维度继续使用 num_key_value_heads = num_kv_heads，不展开到 query heads。

已实施步骤

  1. 新增专用 forward() 路径，避免影响当前 AscendAttentionBackendImpl 和 C8 INT8 路径。
  2. 在 forward() 中强制要求 kv_cache[2:4] 存在，缺失时直接报错，避免静默退化。
  3. 对 query 调用 npu_dynamic_mx_quant，生成 FP8 E4M3 query 和 E8M0 query scale。
  4. decode/cache-hit/chunked-prefill 路径读取 self.key_scale_cache 和 self.value_scale_cache；prefill-no-cache 路径直接量化当前 K/V 并使用即时 scale。
  5. 通过 npu_fused_infer_attention_score_v2 传入 dequant_scale_query、dequant_scale_key、dequant_scale_value、量化模式和 dtype 参数，承接 V5 风格 MXFP8 scale 语义。
  6. 补充精度验证仍待执行：同一输入下比较 BF16 KV baseline、当前 FP8 cache 写入结果和新 MXFP8 attention 输出误差。

风险与待确认

  • 当前 torch_npu 文档主要明确 INT8 KV 伪量化 scale 约束，MXFP8 KV scale 的正式 attention op 入参仍需对齐。
  • 如果现有 FIA/PA 不支持 E8M0 per-32 scale，需要新增自定义 op 或先走显式 dequant 后 attention 的功能验证路径。
  • 全量化 attention 可能要求 query 也量化并提供 query scale，这会扩大修改范围，需要单独设计。
  • 如果算子要求 NZ 或其它 layout，cache 写入布局和 scale layout 可能需要进一步调整。
  • hamming sparse KV compression 当前未接入 C8_MXFP scale cache，后续若需要支持，需要补齐压缩块与 scale cache 的一致性维护。
