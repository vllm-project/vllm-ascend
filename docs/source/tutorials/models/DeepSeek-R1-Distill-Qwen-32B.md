# DeepSeek-R1-Distill-Qwen-32B

## Introduction

`DeepSeek-R1-Distill-Qwen-32B` is a reasoning-oriented large language model distilled by DeepSeek from `DeepSeek-R1` onto the `Qwen2.5-32B` base. It keeps the standard `Qwen2ForCausalLM` (decoder-only, GQA) architecture and is fine-tuned on 800k reasoning samples generated by DeepSeek-R1, which gives it strong chain-of-thought ability on math, code and logical reasoning while remaining a single dense model that vLLM and `vllm-ascend` support natively.

This document describes the main verification steps of the model, including supported features, environment preparation, single-node deployment, functional verification, and accuracy evaluation on the GSM8K benchmark.

This model uses the standard `Qwen2ForCausalLM` architecture, which is natively supported by vLLM and `vllm-ascend`; no custom modeling code or patch is required. It is listed as a supported extended-compatible model (`DeepSeek Distill (Qwen/Llama)`) on Atlas A2/A3 hardware. This example is validated on `vllm-ascend` v0.11.0 and later.

> Note: This is a reasoning model. Its chat template appends `<think>\n` after the assistant turn to force step-by-step reasoning, so responses contain a `<think>...</think>` block. Following the official usage recommendations, do not add a system prompt (put all instructions in the user turn) and use `temperature=0.6`, `top_p=0.95`.

## Environment Preparation

### Model Weight

- `DeepSeek-R1-Distill-Qwen-32B` (BF16 version): the weights are about 65.5 GB (sharded into 8 safetensors files), which exceeds the memory of a single 64 GB card once the KV cache is added. It requires at least 2 Atlas 800I A2 (64G × 2) cards, or 1 Atlas 800 A3 node. [Download model weight](https://www.modelscope.cn/models/deepseek-ai/DeepSeek-R1-Distill-Qwen-32B)

These are the recommended numbers of cards, which can be adjusted according to the actual situation.

It is recommended to place the model weight in a shared cache directory, such as `/root/.cache/`, or a local model path like `/data/vllm-workspace/models/DeepSeek-R1-Distill-Qwen-32B`.

## Deployment

Start the online serving service with the following command. The 32B model is deployed with `tensor_parallel_size=2` so that the weights and KV cache fit across two 64 GB cards:

```bash
vllm serve "deepseek-ai/DeepSeek-R1-Distill-Qwen-32B" \
  --served-model-name deepseek-r1-distill-qwen-32b \
  --tensor-parallel-size 2 \
  --max-model-len 32768 \
  --enforce-eager \
  --gpu-memory-utilization 0.9 \
  --port 8000
```

Key parameter notes:

- `--tensor-parallel-size 2`: splits the 32B weights across 2 NPU cards; increase if you have more cards or hit out-of-memory.
- `--max-model-len 32768`: the maximum generation length recommended by the model card for this reasoning model; the native context ceiling is 131072.
- `--enforce-eager`: matches the model card's recommended launch flags; you can drop it to enable ACL Graph once your environment is verified.

Common Issues Tip: If you encounter startup or environment issues, please refer to the [Public FAQ](https://docs.vllm.ai/projects/ascend/en/latest/faqs.html) for troubleshooting.

## Functional Verification

Once your server is started, you can query the model with a reasoning prompt. Following the model usage recommendations, all instructions are placed in the user turn and no system prompt is used:

```bash
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek-r1-distill-qwen-32b",
    "messages": [
      {"role": "user", "content": "Natalia sold clips to 48 friends in April, and then she sold half as many clips in May. How many clips did she sell altogether in April and May? Please reason step by step, and put your final answer within \\boxed{}."}
    ],
    "temperature": 0.6,
    "top_p": 0.95,
    "max_completion_tokens": 2048
  }'
```

Expected result: an HTTP 200 response whose `choices` field contains a `<think>...</think>` reasoning block followed by the final answer (for this prompt, `\boxed{72}`). The exact text depends on sampling; replace this with the real output captured on your Ascend NPU.

### Offline Inference

```python
from vllm import LLM, SamplingParams

prompts = [
    "The future of AI is",
    "Solve step by step: If a train travels 60 miles in 2 hours, what is its average speed?",
]
sampling_params = SamplingParams(temperature=0.6, top_p=0.95, max_tokens=512)
llm = LLM(
    model="deepseek-ai/DeepSeek-R1-Distill-Qwen-32B",
    tensor_parallel_size=2,
    max_model_len=32768,
    enforce_eager=True,
    gpu_memory_utilization=0.9,
)

outputs = llm.generate(prompts, sampling_params)
for output in outputs:
    print(f"Prompt: {output.prompt!r}, Generated text: {output.outputs[0].text!r}")
```

## Accuracy Evaluation

The GSM8K dataset is used here to evaluate the reasoning capability of `DeepSeek-R1-Distill-Qwen-32B` on Ascend.

The evaluation reuses the e2e config `tests/e2e/models/configs/DeepSeek-R1-Distill-Qwen-32B.yaml`:

```bash
export VLLM_USE_MODELSCOPE=True
pytest -sv tests/e2e/models/test_lm_eval_correctness.py \
  --config ./tests/e2e/models/configs/DeepSeek-R1-Distill-Qwen-32B.yaml \
  --tp-size 2 \
  --report-dir ./benchmarks/accuracy
```

The current evaluation setting is:

- Dataset: `gsm8k`
- Split: `test`
- Few-shot setting: `5-shot`
- `apply_chat_template`: `True`
- `fewshot_as_multiturn`: `True`

The values declared in the config are validated with a relative tolerance of `RTOL=0.05` in `test_lm_eval_correctness.py`. Re-run on your own NPU (or rely on the Ascend nightly CI) and update the threshold values with the measured results if they fall outside the tolerance.

### Remarks on Metrics

- **exact_match,strict-match**: Only predictions that strictly match the expected final-answer extraction format are counted as correct. Reasoning models that emit a `<think>` block can score lower on this stricter rule.
- **exact_match,flexible-extract**: Predictions are evaluated with a more flexible answer extraction rule, tolerating minor formatting differences as long as the final numeric answer is correct.

> DeepSeek's official model card and the DeepSeek-R1 paper (arXiv:2501.12948) report MATH-500 pass@1 `94.3` and AIME 2024 pass@1 `72.6` for this distilled model, but do not publish a GSM8K score. The GSM8K thresholds in the config are therefore reference estimates to be calibrated by measured Ascend results, not official figures.

## Performance

`DeepSeek-R1-Distill-Qwen-32B` can be deployed through `vllm-ascend` for online inference and benchmark evaluation. Actual throughput and latency depend on hardware resources, prompt length, output length, concurrency, and runtime configuration. Because this is a reasoning model that emits long `<think>` sequences, the average output length per request is typically larger than for a non-reasoning model of the same size, which should be taken into account when sizing concurrency.

Refer to [vllm benchmark](https://docs.vllm.ai/en/latest/benchmarking/) for the `vllm bench` subcommands (`latency`, `serve`, `throughput`). Further benchmarking is recommended for:

- request latency
- throughput under concurrency
- long-context inference
- memory utilization
- stability under continuous serving workloads

## FAQ

For common environment, installation, and general parameter issues, please refer to the [Public FAQ](https://docs.vllm.ai/projects/ascend/en/latest/faqs.html); this section only covers model-specific issues.

- **The model does not produce a reasoning block.** Make sure the chat template is applied (use the `/v1/chat/completions` endpoint, not raw `/v1/completions`) so that `<think>\n` is appended to the assistant turn. Do not add a system prompt.
- **Out of memory on startup.** The 32B BF16 weights are ~65.5 GB and do not fit on a single 64 GB card together with the KV cache. Use at least `--tensor-parallel-size 2`, or lower `--max-model-len` / `--gpu-memory-utilization`, or use a quantized weight version.
- **GSM8K accuracy differs from the config thresholds.** The thresholds are estimates pending Ascend CI calibration. If the measured value is outside `RTOL=0.05`, update the config with the measured value and note the software stack (CANN / torch_npu / vLLM / vllm-ascend versions) in your PR.
